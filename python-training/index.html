Day 1 | Pandas

Inspecting your data: the first five checks

Every time you load a new file, run these five checks before doing anything else.

txns.head()               # 1. Look at the first 5 rows
txns.shape                # 2. How many rows and columns?
txns.dtypes               # 3. What type is each column?
txns.isna().sum()         # 4. How many missing values per column?
txns["txn_id"].nunique()  # 5. How many unique IDs? (matches row count?)

If the number of unique IDs does not match the row count, you may have duplicates.

Day 1 | Pandas

Filtering rows

Use a condition inside square brackets to keep only the rows you want.

# Only posted transactions
posted = txns[txns["status"] == "POSTED"]

# Transactions above 1000 SAR
large = txns[txns["amount_sar"] > 1000]

# Combine conditions with & (and) or | (or)
mobile_large = txns[
    (txns["channel"] == "MOB") & (txns["amount_sar"] > 1000)
]

Always wrap each condition in parentheses when combining with & or |.

Day 1 | Data quality

Missing values: what .isna() tells you

Missing values appear as NaN in pandas. They are common in banking data and must be understood before analysis.

# Count of missing values per column
txns.isna().sum()

# Percentage of missing values per column
null_rates = txns.isna().mean().sort_values(ascending=False)
null_rates
Rule of thumb

A column with more than 30% missing values needs investigation. Ask: is this field optional, or is something wrong with the extract?

Day 1 | Data quality

Finding duplicate records

Duplicates can inflate counts and distort KPIs. Always check for them before aggregation.

# Are there any duplicate transaction IDs?
txns["txn_id"].duplicated().sum()

# Show all rows with a duplicated ID
dupes = txns[txns.duplicated("txn_id", keep=False)]
dupes

keep=False marks all copies as duplicates, not just the second one. This helps you see the full picture.

Day 1 | Data quality

Data types matter: why IDs should be strings

A branch ID like 001 looks like a number, but if pandas reads it as an integer, it becomes 1 and you lose the leading zero. This breaks joins later.

# Read with explicit types to protect IDs
txns = pd.read_csv(
    "data/transactions.csv",
    dtype={"account_id": "string", "branch_id": "string"}
)

# Convert dates properly
txns["txn_ts"] = pd.to_datetime(txns["txn_ts"], errors="coerce")

errors="coerce" turns unparseable dates into NaN instead of crashing.

Day 1 | Functions

Why write functions?

A function packages logic so you can reuse it and so other people can understand what it does.

def clean_amount(text):
    """Convert a text amount like '1,250.00' to a float."""
    if pd.isna(text):
        return None
    cleaned = str(text).replace(",", "").strip()
    if cleaned == "":
        return None
    return float(cleaned)

A good function name tells you what it does. A good docstring tells you how to use it.

Day 2 | Joins

What is a join?

A join combines two tables using a shared key column. It is how you connect customers to their accounts, or transactions to their branches.

Customers

customer_id, name, region, segment

Accounts

account_id, customer_id, product_family

Transactions

txn_id, account_id, amount, channel, branch_id

The key columns (customer_id, account_id) are what connect these tables together.

Day 2 | Joins

Join types in pandas

# Left join: keep all rows from the left table
result = txns.merge(accounts, on="account_id", how="left")
Left

Keep all left rows. Fill with NaN if no match on the right.

Inner

Keep only rows that match in both tables.

Outer

Keep all rows from both tables. Fill gaps with NaN.

Which to use?

Left join is safest. You keep all your analysis rows and can see where matches are missing.

Day 2 | Joins

Protecting the grain

The grain of a table is what one row represents. If you join a customer table (one row per customer) to a transactions table (many rows per customer), the result has one row per transaction. That is correct.

But if customers has duplicate IDs, the join will create extra rows. Always check:

# Before joining, verify the key is unique
customers["customer_id"].duplicated().sum()
# If this is not 0, investigate before proceeding.
Day 2 | Joins

Building a lean analysis table

Join only the columns you need. Lean tables are easier to work with and harder to misinterpret.

analysis = (
    txns
    .merge(accounts[["account_id", "customer_id", "product_family"]],
           on="account_id", how="left")
    .merge(customers[["customer_id", "region", "segment"]],
           on="customer_id", how="left")
    .merge(branches[["branch_id", "city"]],
           on="branch_id", how="left")
)

Day 2 | GroupBy

Introduction to groupby

groupby is how you summarise data. It follows a three-step pattern: split the data into groups, apply a calculation, and combine the results.

# Count transactions per branch
txns.groupby("branch_id")["txn_id"].count()
1. Split

Divide the data by the grouping column (e.g. branch_id).

2. Apply

Run a calculation on each group (e.g. count, sum, mean).

3. Combine

Bring the results back into a single table.

Day 2 | GroupBy

Multiple aggregations with .agg()

Use .agg() to compute several metrics at once. Name each output clearly.

branch_kpi = (
    posted
    .groupby(["region", "branch_id"])
    .agg(
        txn_count     = ("txn_id",     "count"),
        total_fee_sar = ("fee_sar",    "sum"),
        avg_ticket    = ("amount_sar", "mean"),
    )
    .reset_index()
)

branch_kpi.head()

.reset_index() turns the group keys back into regular columns.

Day 2 | KPIs

Defining metrics clearly

Every KPI has three parts. If you cannot state all three, the metric is not ready for leadership.

Numerator

What are you counting or summing?

Example: sum of fee_sar

Denominator

What is the base or population?

Example: per branch, for posted transactions only

Filter

What rows are included or excluded?

Example: status == "POSTED", exclude test accounts

Day 2 | Shares

Computing shares and percentages

Shares show proportions. The denominator you choose changes the story.

# Count accounts by region and product family
uptake = (
    accounts
    .merge(customers[["customer_id", "region"]], on="customer_id")
    .groupby(["region", "product_family"])
    .size()
    .reset_index(name="count")
)

# Compute share within each region
uptake["share"] = (
    uptake["count"] / uptake.groupby("region")["count"].transform("sum")
)
Day 2 | Shares

Choosing the right denominator

The same numerator with different denominators tells different stories.

Denominator: all accounts in the region

"What share of accounts in the Eastern region are savings accounts?"

Answers: product mix

Denominator: all savings accounts nationally

"What share of all savings accounts are in the Eastern region?"

Answers: regional concentration

Both are valid. Which one you choose depends on the question leadership is asking.

Day 3 | ML handoff

Feature engineering: what and why

The Machine Learning module that follows this programme needs a customer-level table of features. Your job is to prepare data that a model can learn from.

Feature

A column that describes past customer behaviour. Examples: transaction count, average ticket size, digital channel share.

Cut-off date

The date that divides "what happened" from "what you are trying to predict". Features must only use data before the cut-off.

Leakage

When future information accidentally appears in your features. This makes the model look good in testing but fail in production.

Day 3 | ML handoff

Building a feature table

cut_off = pd.Timestamp("2026-06-30")
window = posted[posted["txn_ts"] <= cut_off].copy()
window["is_digital"] = window["channel"].isin(["MOB", "ONLINE"])

features = (
    window.groupby("customer_id")
    .agg(
        txn_count    = ("txn_id",        "count"),
        avg_ticket   = ("amount_sar",    "mean"),
        digital_pct  = ("is_digital",    "mean"),
        total_fee    = ("fee_sar",       "sum"),
        product_cnt  = ("product_family", "nunique"),
    )
    .reset_index()
)
features["cut_off_date"] = "2026-06-30"
features.to_csv("outputs/day3_pack/features.csv", index=False)
Day 3 | ML handoff

Avoiding data leakage

Leakage is the most common mistake in feature engineering. Here are two ways it happens:

Time leakage

Using transaction data from July to predict what happens in June. The model sees the future.

Fix: enforce a hard cut-off date.

Target leakage

Including a column that is only available after the outcome is known (e.g. "churned" flag in a churn prediction model).

Fix: only include features available at prediction time.

Day 3 | Governance

Data governance: what to export and what not to

Under Saudi Arabia's Personal Data Protection Law (PDPL), you must be intentional about what data leaves your environment.

Good practice
  • Export only the fields needed for the agreed output
  • Document what is included and why
  • Use pseudonymised identifiers
Risky practice
  • Export everything "just in case"
  • Include personal names or national IDs in feature tables
  • No documentation of assumptions

Your task

In teams, assemble your strongest outputs into a compact pack and present a 90-second executive brief.

What to prepare
  • Choose your strongest evidence
  • Assemble a compact pack
  • Write a three-sentence summary
  • Prepare to present live
Presentation frame
  1. Open with the most important finding
  2. Show the one chart or table that proves it
  3. State the likely driver
  4. Name the key caveat
  5. Finish with the recommended action
